現状の問題と、基盤が解決すること

今の問題

  • GMO API の15分足に依存 → 7分足や3分足でテストできない
  • 取得できる過去データが限定的 → 長期の検証ができない
  • スクリプトがベタ書き → 60パターンの一括比較ができない
  • 「この戦略が一番いい」をデータで証明できない

基盤で解決すること

  • 任意の時間足（1分〜15分、7分も3分も）
  • 20年分のデータで長期検証
  • 60パターン一括実行 → ランキングで比較
  • 本番と同じ Rule をそのまま使える

ParameterSet

ただのデータ。Runner だけが読む。他の部品には渡さない

RUNNER

Runner — 司令塔

ParameterSet を読む唯一の存在。全部品を組み立てて実行する

Runner がやること

1. ParameterSet（60セット）を読む

2. 各セットごとに Engine + Rule + Simulator を生成

3. Engine を実行 → BacktestResult が返る

4. 全結果を ResultStore に保存

なぜ Runner だけが ParameterSet を読むのか

Engine や Rule にパラメータを直接渡すと、各部品が「自分に関係ない設定」まで見えてしまう。
Runner が読んで、各部品が必要とする型に変換して注入する。情報の漏洩を防ぐ設計。

ParameterSet の例

{ timeframe: '7m',

  pair: 'USD_JPY',

  strategy: 'SMA_CROSS',

  shortPeriod: 25,

  longPeriod: 75,

  dateFrom: '2006-01-01',

  dateTo: '2026-03-31' }

Runner が変換する先

EngineConfig timeframe, pair, dateRange

Rule 戦略パラメータ注入済みインスタンス

Simulator Ideal or Realistic

DataProvider TimescaleDataProvider

ENGINE

Engine — 実行ループ

足を1本ずつ処理する。自分では判定も約定もデータ取得もしない

Engine のループ（擬似コード）

// 1. データ取得

candles = dataProvider.getOHLC(config)

// 2. 足ごとにループ

for (candle of candles) {

  // 3. Snapshot 組立

  snapshot = adapter.build(candle)

  // 4. Rule に判定させる

  command = rule.shouldEntry(snapshot)

  // 5. 約定シミュレーション

  result = simulator.simulateEntry(command)

  // 6. ポジション管理

  position = Position.open(command, result)

  positions.set(result.id, position)

}

Engine が知っていること

EngineConfig（期間・足・ペア）と、4つのインターフェースだけ。
DB の種類も、戦略の中身も、約定のやり方も知らない。

Engine が知らないこと

- ParameterSet の存在

- Rule が SMA なのか RSI なのか

- データが TimescaleDB から来ているのか CSV か

- 約定が理想なのか現実なのか

Phase 1 と Phase 2 で同じコード

違いは Runner が注入する DataProvider と Simulator の実装だけ。Engine の if 分岐はゼロ。

RULE

Rule — 判定だけ

本番の Rule をそのまま使う。BT 用に書き換えない

Rule のインターフェース

interface EntryRule {

  shouldEntry(snapshot: MarketSnapshot)

    : EntryCommand | DoNothing

}

interface ExitRule {

  shouldExit(snapshot, position)

    : ExitCommand | DoNothing

}

なぜ本番のコードをそのまま使うのか

BT 用に書き換えると「BT では勝つが本番では負ける」が起こる。
同じコードで検証するからこそ、BT の結果が本番に直結する。

4つの戦略

SMA

短期・長期 SMA のクロスで売買

RSI

買われすぎ・売られすぎで逆張り

乖離

SMA からの乖離率で逆張り

ヒゲ

ローソク足のヒゲの長さで逆張り

Rule は何を知っているか

MarketSnapshot（市場データ）だけ。Engine も Runner も BT の存在すら知らない。

SnapshotAdapter のおかげで

Rule は「今、本番で動いている」のか「20年前のデータでテストされている」のか区別できない。これが正しい設計。

EXECUTION SIMULATOR

ExecutionSimulator — 約定の再現

Issue #70 | 「どの価格で約定したか」をシミュレートする

IdealExecutionSimulator（Phase 1）

足 N+1 の open で即約定

スプレッドなし、スリッページなし

PositionId は UUID で採番

60パターンを高速に回すためのシンプル版

RealisticExecutionSimulator（Phase 2）

買いは ask、売りは bid で約定

スリッページ: 0〜0.5pips のランダムズレ

約定遅延: 50〜200ms 後の tick で約定

最終候補を現実に近い条件で検証

なぜ分けるのか

Phase 1 で Realistic を使うと遅すぎる（60パターン x 20年 x tick）。
Phase 1 で候補を5つに絞ってから、Phase 2 で精密検証する。

Engine のコード（両 Phase で同一）

result = simulator.simulateEntry(cmd, data)

position = Position.open(cmd, result)

positions.set(result.positionId, position)

Engine は Ideal か Realistic かを知らない。Runner が注入を切り替えるだけ。

Phase 1 の限界

最重要

時間整合性 — バイアス防止

これが崩れるとバックテスト結果が全部嘘になる

ルックアヘッドバイアスとは

「未来の情報を使って過去の判定をする」こと。
例: 15時の足の close で「買いだ」と判定して、同じ 15時の close で約定する。
→ 現実では close の値は足が確定するまでわからない。カンニングと同じ。

正しい順番（本番と同じ）

1. 足 N が確定する

2. 足 N の confirmed（SMA 等）で Rule が判定

3. Rule が「買い」と言う

4. 足 N+1 の open（= 次の最初の価格）で約定

判定に使った足の close で約定しない。これが鉄則。

Phase ごとの約定タイミング

Phase判定約定価格
Phase 1足 N の confirmed足 N+1 の open
Phase 2足 N の confirmed足 N 確定後の最初の tick

テストで検証すること

- 約定価格が判定足の close と異なること

- 約定時刻が判定足の時刻より後であること

- warmup 期間中にトレードがないこと

安全装置

Warmup 期間

インジケーターが有効になるまで Rule を呼ばない

問題

SMA(100) は100本の確定足がないと有効な値を出せない。
最初の99本で Rule を呼ぶと、SMA の値が不正確で偽シグナルが連発する。

対策

1. DataProvider が date_from より前のデータも warmup 分だけ余分に取得

2. SnapshotAdapter はインジケーター計算に全足を使う

3. Engine が Rule を呼び始めるのはwarmup 完了後から

4. warmup 足数は EngineConfig に含める
