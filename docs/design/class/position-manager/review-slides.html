背景 — なぜ今やるか

今のシステムは「1 つの戦略だけ動く」ように作られています。4 戦略を同時に動かすには、中心に司令塔を置く必要があります。

🔴 今の状態

  • SMA クロス戦略だけが動いている
  • 同じ通貨ペアではポジションは 1 つまでしか持てない
  • ロットを計算する時の「元手」は CAPITAL 環境変数で固定値（例: 10 万円）として使っている
  • 戦略が 1 つなので、複数シグナルの調整ロジックは不要だった

🟢 これからやりたいこと

  • 4 つの戦略が同時に動く（SMA クロス / RSI 逆張り / SMA 乖離 / ヒゲ）
  • 戦略ごとに別々のポジションを持てる
  • ロットは実際の口座残高を見て動的に決める
  • 複数戦略が同時にシグナルを出したときに、まとめて賢く捌く司令塔を置く
👉 この「司令塔」が今回作る PositionManager。
「どの戦略が」「どの順で」「どれくらいのロットで」発注するかを決めてくれる中心部品です。

※ 新しい戦略そのもの（RSI 逆張り等）の実装は今回は含みません。PositionManager の「受け口」だけ作って、戦略が揃った時にはめ込めるようにするのが狙いです。

全体像 — 層のどこに何を置くか

Luchida は「Clean Architecture」と呼ばれる構造でできています。役割ごとにフォルダ（層）を分けて、互いの依存方向を制御します。

層ごとの性格（超ざっくり）

  • domain（ドメイン） — ビジネスの計算と判断だけ。外の世界（DB・API）を一切知らない
  • application / action — 「ユースケース」の組み立て役。domain を呼んで処理を繋げる
  • port — 外とのやり取りの「受け口」の定義（抽象インターフェース）
  • infrastructure / adapter — 実際に DB や API を叩く実装

今回、各層に追加・変更するもの

層追加するもの
domain ロジック: LotPolicy interface / MaintenanceRatioBasedLotPolicy 実装 / AllocationPolicy interface / EqualWeightAllocationPolicy 実装
データの箱: LotAllocation, Ratio, Balance, Money, Rate, MaintenanceRatio, MarginRate, LotDecisionInput, AllocationContext
application PositionManager（司令塔） / PositionSizingService（ロット計算の組み立て）/ EntryQueue（発注の順番制御）
port BalancePort — メソッド 2 つ:
・current() は残高を返す（5 秒キャッシュ可。失敗時は null）
・freshNow() は発注直前用。キャッシュを無視して最新を取得
infrastructure GmoBalanceAdapter — 実際に GMO API を叩く。内部で 5 秒のキャッシュを持ち、API 失敗時は CAPITAL を返す

中心の 3 つの部品

この 3 つが今回の主役。それぞれの役割を比喩つきで説明します。

🎯 PositionManager

application 層 / 今回新しく作る

複数の戦略（Rule）が「買いたい」「売りたい」と報告してくる。それを取りまとめて、誰がいくらずつ買うかを決めて、順番に発注する役目。

🏗️ 比喩: 現場監督。複数の作業員の声をまとめて、今日の分担を決める。

⚖️ AllocationPolicy

domain 層 / ドメインサービス（ロジックの固まり）

「今どの戦略が反応していて」「いくらポジションを持っていて」「残高はいくら」から、戦略ごとの配分比率を決めるロジック。

interface なので実装を差し替え可能。初期実装は均等配分（全戦略に同じ比率）。

🍕 比喩: ピザを切り分けるルール。ルールは差し替え可能（4 等分 / 差をつける / etc.）。

📦 LotAllocation

domain 層 / 値オブジェクト（データの箱）

AllocationPolicy の計算結果。「A: 0.7、B: 0.3」のような比率の組み合わせを表す不変のデータ。

内部で「合計は必ず 1.0」というルールを守る。

🍕 比喩: 切り分けられた後のピザ。数字の組み合わせ。それ以上でもそれ以下でもない。
重要なポイント: 「配分を決めるロジック (Policy)」と、「配分の結果のデータ (Allocation)」を分けている。
この分離がこの設計の要です。次のスライドでもう少し詳しく説明します。

StrategyWeight をやめた理由

最初は StrategyWeight（戦略の重み）という 1 つの名前で扱おうとしていました。DDD レビューで「中身が混ざってる」と指摘され、2 つに分けました。

❌ Before: StrategyWeight 1 個で扱う

  • 「戦略の重み」という曖昧な概念
  • 数字の箱（比率 0.7）とロジック（その比率の決め方）が 1 つのクラスに同居
  • 状態依存のロジック（例:「A 保有中なら B を半減」）を足そうとすると、データの箱の中に外部状態が漏れて汚くなる
  • 配分ロジックだけ差し替えたくても、データの箱ごと触ることになる

✅ After: 2 つに分ける

  • LotAllocation = 計算結果のデータ（不変・副作用なし）
  • AllocationPolicy = 計算するロジック（interface で差し替え可能）
  • ロジックを変えたいときは Policy の実装を差し替えるだけ。LotAllocation は触らない
  • 将来 ML ベース / ConvictionScore ベースなど新しい実装が来ても同じ枠で扱える
増田亨ペルソナからの指摘（意訳）:
「値オブジェクトは属性値だけで同一性が決まる不変の塊。外部状態（他のポジションを保有中かどうか等）を参照した瞬間にそれは値ではない」
→ だから、値（データ）とロジック（判断）を分けよう、という話。

※ 値オブジェクト とは: ただの数値ではなく、意味と制約を持ったデータの箱。例えば Balance は「負になれない」「通貨単位を持つ」といった制約を内部に持っている。

初期実装は「均等配分」から始める

ユーザーから「A+B なら 7:3、A+B+F なら 1:4:5」という具体的な配分要望がありましたが、金融アナリストのレビューで筋が悪いと判定されました。

固定比率の問題点

  • 「7:3 にする根拠は何?」と問われたときに答えが難しい。大抵は感覚値になる
  • SMA クロス / RSI / SMA 乖離 / ヒゲ は全部同じ通貨ペアの短い時間足で動くので、値動きが似ている。特に逆張り 3 兄弟はほぼ同じ場面で反応する
  • 組み合わせの数が爆発する（4 戦略で 15 通り、5 戦略で 31 通り、すべての設定を用意するのは非現実的）
  • 「A 保有中は B 半減」のような離散的なルールは、過去データに合わせすぎ（カーブフィット）の温床になりやすい

均等配分の根拠

  • 「情報がないときの最善手」として認められている方法（DeMiguel et al., 2009 の有名な論文）
  • 実装がシンプルで、説明しやすく、バグも少ない
  • 後で複雑な配分ロジックを入れたくなっても、AllocationPolicy の実装を差し替えるだけ。既存の呼び出し側を変えなくていい
  • 将来的には「リスクパリティ」「ConvictionScore 重み付け」などに置き換える想定
スタート地点は単純さ優先。複雑な配分ルールは本当に効果があるか分からないまま複雑化させず、まずはデータを取りながら少しずつ洗練させる方針です。

※ 組み合わせベース・相関補正・リスク予算ベースの実装は別 issue（#120 / #121）に切り出しています。

CAPITAL を予備値に変える理由

今のロット計算は「口座は 10 万円あるつもり」で動いています。実残高と離れていく一方です。これを直します。

今のコード（簡略化）

const CAPITAL = 100000;  // 起動時に 10 万円と決め打ち
const lotPolicy = new MarginBasedLotPolicy(
  () => CAPITAL,     // ← いつも 10 万円を返す固定関数
  ...
);

これの何が問題か

実際の残高CAPITAL 固定値ロット計算に使う値起きる問題
10 万円（そのまま）10 万円10 万円問題なし
7 万円（損失で減った）10 万円10 万円 実残高より大きいロットで発注 → 設計より大きなリスク
13 万円（利益で増えた）10 万円10 万円 ロットが増えない → 複利効果ゼロ
特に危険なのは損失局面。
残高 7 万円なのに 10 万円前提で維持率 140% を狙ってロットを出すと、実際の維持率は 98% 程度になり、GMO 側の強制決済ライン（100%）を割る可能性があります。

目指す状態

  • 毎回のロット計算で、その時点の口座残高を GMO API から取得して使う
  • API が失敗したとき（メンテナンス・通信障害など）のみ、予備値として CAPITAL を使う
  • つまり CAPITAL は「滅多に使わない非常用の値」になる

最初の 4 案を全部却下して案 E を採用した話

残高取得をコードのどこに置くか、最初は 4 案考えました。DDD レビューで全部却下され、案 E（5 番目）が採用されました。

案やりかた却下された理由
A 残高を返す関数をひそかに注入 関数の名前は getCapitalJpy（元手を返す）なのに中身は残高を取得。名前が嘘をついている。読んだ人が必ず誤解する
B ロット計算クラスに残高取得を統合 ロジックと I/O（外部との通信）が混ざる → テストできない・再利用できない

E（採用） LotPolicy は純粋ドメインサービスに戻し、application 層に組み立て役を新設 責務がきれいに分かれる。テストしやすい。将来の拡張に耐える

class PositionSizingService {
  execute(pair) {
    // ① 残高取得（失敗したら予備の Balance を使う）
    const balance = this.balancePort.current() ?? this.fallbackBalance;
    // ② レート取得
    const rate = this.ratePort.currentOf(pair);
    // ③ 必要な値を束ねて LotPolicy に渡す（純粋計算）
    return this.lotPolicy.decide(
      new LotDecisionInput(pair, balance, rate, MaintenanceRatio.of(1.4))
    );
  }
}

CAPITAL は main.ts で Balance.of(Money.jpy(CAPITAL)) のようにデータの箱に包んで、PositionSizingService のコンストラクタに「予備残高」として注入します。

部品の名前を整理した話

名前はユビキタス言語（チームで共通の言葉）。読んだ人が誤解しないよう、今回 3 つをリネームしました。

変更前変更後理由
DynamicLotCalculator PositionSizingService 「Dynamic（動的）」は実装上の都合で、使う人から見たら「普通」のこと。わざわざ付ける必要がない。
「Calculator（電卓）」は軽く聞こえる。FX 業界では "Position Sizing"（ポジションサイズをどう決めるかの判断）という言葉が定着しているので、そっちに合わせた。
MarginBasedLotPolicy MaintenanceRatioBasedLotPolicy 「Margin（証拠金）ベース」だと範囲が広い。実態は「維持率が目標 140% になるよう決める」ロジックなので、名前で意図がわかるようにした。
BalanceCache GmoBalanceAdapter 「Cache（キャッシュ）」は実装手段を名前に出してしまっている。役割は「GMO から残高を取る窓口」なので、Adapter（つなぐ部品）と呼ぶ方が自然。キャッシュはその中の実装詳細。
StrategyWeight LotAllocation + AllocationPolicy 前のスライドで説明した通り、値（比率データ）と判断（決めるロジック）を分けた結果、名前も 2 つに分かれた。
命名は「将来コードを読む人（他人でも 3 ヶ月後の自分でも）」のために選ぶ。名前で役割が伝わるなら、コメントを書かずに済む。

値オブジェクトを整備する話

このプロジェクトでは「domain 層に number 型を置かない」という方針があります。今回それを本気で徹底します。

値オブジェクトって何?

ただの数値ではなく、意味とルールを内側に持ったデータの箱。

  • 例: Balance（残高）は「負になれない」というルールを内側に持つ
  • 例: Ratio（比率）は「0 から 1 の範囲」
  • 例: MaintenanceRatio（維持率）は「1.0 より大きい」

数値（number）で扱っていると、うっかり負の残高や範囲外の比率をコードに渡してしまう。それを値オブジェクト自身が防ぐのがポイント。

今回追加する値オブジェクト一覧

名前役割・内部ルール
Balance口座残高。非負。内部に Money を持つ
Money金額と通貨のセット。数値と単位を分離
CurrencyJPY / USD などの通貨種別
Rateレート。null を返さない（取得失敗は例外にして発注を中止）
MaintenanceRatio維持率の目標値（例: 1.4 = 140%）。1.0 より大きい必要あり
MarginRate証拠金率（例: 0.04 = レバレッジ 25 倍）。(0, 1) の範囲
Ratio0〜1 の比率。LotAllocation の内部で使う
LotDecisionInputLotPolicy 呼び出し時の引数をまとめた箱（パラメータオブジェクト）
LotAllocation配分結果。合計 = 1.0 のルール。Map を外に出さず ratioOf(s) で問い合わせる

👉 現状 number で引き回している変数は、このリファクタですべて値オブジェクトに置き換えます。既存テストも書き直し。

EntryQueue（発注の信号機）

GMO FX の API には「発注は 1 秒に 1 件まで」という制限があります。4 戦略が同時に発注したいと言ってきた時、この制限を守る仕組みが必要です。

🚦 比喩: コンビニのレジ。複数のお客さんが一斉に来ても、レジは 1 人 1 人順番に対応する。

複数ポジションの扱いかた

今は「同じ通貨ペアでポジションは 1 つまで」という制約があります。4 戦略同時稼働のために、「戦略ごとに 1 つまで」に緩和します。

決めたこと

DB のテーブル positions に、「通貨ペア × 戦略名 の組で OPEN 状態のレコードは 1 つまで」という制約を追加する。

✅ 許可するケース

  • USD/JPY × SMAクロス = 1 件 OPEN
  • USD/JPY × RSI 逆張り = 1 件 OPEN
    （SMAクロスと同時に存在しても OK）
  • CLOSED（決済済み）のポジションは何件あっても OK

❌ 禁止するケース

  • USD/JPY × SMAクロス = 2 件 OPEN
    （同戦略で複数持つのは禁止）

将来「同戦略でドテン（途転）を許したい」等があれば、その時に制約を緩めればいい。

設計の原則: 制約は最初に厳しく、必要に応じて緩めるのが安全。逆（緩いところから厳しく）は既存データとの整合性で詰みやすい。

技術的な実装

-- PostgreSQL の「部分ユニークインデックス」
CREATE UNIQUE INDEX idx_positions_open_unique
  ON positions (pair, strategy_name)
  WHERE status = 'OPEN';

ExitRule（決済ルール）の評価ループも「pair, strategy_name でポジションを引いて戦略ごとに評価」する形に変わります。

エントリー全体の流れ

ここまでの部品が、実際にどう連携するのか。15 分足が確定した瞬間から発注までの流れを追います。

  1. 15 分足が確定 → TradingSession が登録されている各 EntryRule に「シグナルある?」と順番に聞く
  2. 「あり」と返した Rule たちを PositionManager が取りまとめる（検知シグナル集合 + 今持ってるポジション + 現在残高もセットで）
  3. PositionManager が AllocationPolicy.decide() を呼ぶ
    → 初期実装は均等配分。戦略数で分ける
  4. 返ってきた LotAllocation（比率の箱）を使って、戦略ごとに PositionSizingService.executeWithFresh(pair) で SizingResult（lot, rate, requiredMargin）を取得
    → 残高ベースで「配分比率 × 基準 Lot」を算出。発注直前の鮮度保証レートを使う
  5. 戦略ごとに EntryCommand（requiredMargin 込み）を組み立てて EntryQueue.enqueue() に投入
  6. EntryQueue が EntryExecution を呼ぶ。POST 1 秒 1 件の制限は GmoRestClient.throttlePost 側で守る
    → 3 秒経った古いシグナルは破棄
  7. EntryExecution が Broker.placeEntry() で GMO API に発注 → 成功したら PositionRepository.register() で DB に保存

TradingGuard（将来の監視役）との境界

将来 TradingGuard という「取引の番人」を作る予定です。AllocationPolicy と責任が被らないよう、今のうちに線を引いておきます。

TradingGuard って何?

  • 経済指標発表中、API メンテ中、異常検知時などに全戦略を止める
  • 月利 20% 到達で「慎重モード」に切り替えるなど、経営判断の番人
  • 他の判断（AllocationPolicy 等）を握りつぶせる最大権力を持つ

今回は実装しないが、将来確実に作る想定。

なぜ今これを気にするのか?

AllocationPolicy（配分ロジック）と TradingGuard（全停止役）は、将来どちらも「ある条件で発注を抑制する」という機能を持ちます。今のうちに責任範囲を分けておかないと、後で「どっちがやるの?」で揉めるし、両方に同じロジックが書かれてしまう危険があります。

引いた線

TradingGuard（将来）AllocationPolicy（今回）
何のために? 市場環境起因の「全停止」 取引可能な状態の中で「配分調整」
きっかけ 経済指標・API メンテ・異常検知・月利 20% 到達など 検知されたシグナル集合 + 現ポジション + 残高
出すもの 「取引可能 / 不可」の 2 値 + 既存ポジションの強制決済指示 LotAllocation（戦略ごとの配分比率）
権力の強さ 最大（他を握りつぶせる） TradingGuard が許可した範囲内で調整するだけ
将来「A 保有中は B を半減」のような抑制ルールを追加するとき:
・市場環境起因（日銀会合中は全停止 等）→ TradingGuard
・戦略組合せ起因（A 保有中は B を半減 等）→ AllocationPolicy の SuppressionRule（別 issue #120）

レビューで見てほしい観点まとめ

承認する前に、次の 4 視点でざっと確認してください。

🎯 スコープ

  • PositionManager + 周辺基盤に絞った判断は妥当か
  • DD 可視化・固定比率・リスク予算・LotCap を別 issue に分離した判断に違和感ないか
  • 今回やらないことが明確になっているか

🏗 設計

  • LotAllocation（値）と AllocationPolicy（判断）の分離は自然か
  • 案 E（LotPolicy 純粋 + PositionSizingService 組み立て）は読みやすい構造か
  • 値オブジェクトとして用意した範囲は十分か

📛 命名

  • PositionSizingService / MaintenanceRatioBasedLotPolicy / GmoBalanceAdapter は読んで意図が分かる名前か
  • ユビキタス言語として違和感はないか
  • 他の部分と一貫性があるか

🔮 拡張性

  • 新しい戦略を追加するときに、設定ファイル変更だけで済むか
  • ConvictionScore (#57) 導入時に AllocationPolicy を差し替えるだけで済むか
  • TradingGuard 実装時に責務境界が保たれるか
