2a. <<interface>> とは

<<interface>> って何?

interface（インターフェース） とは、「こういうメソッドを持ちますよ」という契約だけを定義したもの。中身の実装は持ちません。

<<interface>>
Broker
+ placeEntry()
+ placeExit()
←──── implements ────
GmoBrokerAdapter
(実装クラス)

上の図は「Broker という interface に対して、GmoBrokerAdapter がその実装を提供している」という関係を表しています。

なぜ interface を使うのか?

主に 差し替え可能にするため です:

  • テストでは「本物の GMO に発注しない偽物の Broker」を使いたい
  • 将来 GMO 以外のブローカー（例: OANDA）にも対応するかもしれない
  • 使う側（例: EntryExecution）は「Broker という契約」だけ知っていればよく、どの実装が繋がるかは気にしない

コードで書くとこんな感じ:

// 契約だけ定義
interface Broker {
  placeEntry(command: EntryCommand): Promise<EntryResult>;
}

// 本物の実装
class GmoBrokerAdapter implements Broker {
  placeEntry(cmd) { /* 実際の GMO API コール */ }
}

// テスト用の偽物
class FakeBroker implements Broker {
  placeEntry(cmd) { /* メモリに記録するだけ */ }
}

// 使う側は interface を通して呼ぶ
class EntryExecution {
  constructor(private readonly broker: Broker) {}  // ← 契約で受ける
  execute(cmd) { this.broker.placeEntry(cmd); }
}

「interface で受ける」ことで、使う側のコードは変えずに、本物と偽物を切り替えられます。

2b. 矢印の 4 種類

クラス図に出てくる矢印は、意味が 4 通りあります。形と色を覚えると意味がすぐわかります。

─ ─ ─▷
実装関係（implements） — 点線 + block 矢印。「A interface を B クラスが実装している」
─────▷
継承（extends） — 実線 + block 矢印。「A クラスを B クラスが継承している」（本設計ではあまり使っていません）
─ ─ ─ ➤
依存（uses） — 点線 + open 矢印。「A が B を使う（呼ぶ・引数に取る）」
◆─────
合成（composition） — 菱形 + 実線。「A は B を内部に持っている（B 無しでは A が成立しない）」

例で見る

composition-entry-flow.drawio の中から:

  • PositionManager ─ ─ ─➤ AllocationPolicy — PositionManager は AllocationPolicy を使う（内部にインスタンスを持って呼ぶ）
  • GmoBalanceAdapter ─ ─ ─▷ BalancePort — GmoBalanceAdapter は BalancePort を実装する（契約を満たす）
  • LotDecisionInput ◆──── Balance — LotDecisionInput は Balance を内部に持つ（フィールドとして保持）

3. 層（レイヤー）の考え方

Luchida は Clean Architecture という設計パターンに沿っています。コードを役割ごとにフォルダ（層）に分けて、互いの依存方向をルール化しています。

domain（ドメイン）
ビジネスの計算と判断だけ。DB も API も知らない純粋なロジック
application (action)
「ユースケース」の組み立て役。domain を呼んで処理を繋げる
port
外の世界との「受け口」の定義（interface のみ）
infrastructure / adapter
実際に DB や API を叩く実装

3a. domain（ドメイン）

ビジネスロジックだけが住む場所。外の世界（DB、API、ファイル、時刻、ネットワーク）を一切知らないように作ります。

ここに置くもの:

  • 値オブジェクト（Balance、Rate、Lot など）
  • エンティティ（Position など）
  • ドメインサービス（LotPolicy の実装、AllocationPolicy の実装）

なぜ純粋に保つのか? テストが簡単になる、再利用しやすい、変更の影響範囲が限定される、など複数の理由があります。

3b. application (action)

ユースケースの組み立て役。「〇〇をしたい」という要求を受けて、domain の部品を順番に呼んで処理を完結させます。

ここに置くもの:

  • PositionManager — 「シグナルを受けて発注する」ユースケース
  • PositionSizingService — 「残高からロットを決める」ユースケース
  • EntryExecution — 「個別エントリーを実行する」ユースケース
  • EntryQueue — 発注順序を管理する部品

application 層は domain を呼び出し、port に対して（実装クラスではなく）依頼を出します。

3c. port

外の世界との「受け口」の定義。interface だけを置く場所。実装は置きません。

ここに置くもの:

  • BalancePort — 残高取得の契約
  • RatePort — レート取得の契約
  • Broker — 発注の契約
  • PositionRepository — ポジション保存の契約

port があるおかげで、application は「残高を取る方法」を知らずに「残高を取って」とだけ頼める。実装は infrastructure 層が提供します。

Ports & Adapters パターンと呼ばれる考え方です。core（domain + application）は「穴」（port）だけ空けておき、そこに「プラグ」（adapter）を差し込む構造。プラグ（例: GMO or OANDA）を差し替えても core は影響を受けません。

3d. infrastructure / adapter

実際に DB や API を叩く実装。port の契約を満たす具体的なクラスを置きます。

ここに置くもの:

  • GmoBalanceAdapter — GMO API で残高取得。内部で 5 秒キャッシュ
  • GmoBrokerAdapter — GMO API で発注
  • PgPositionRepository — PostgreSQL でポジションを保存

この層は「どうやって外部と話すか」を全部引き受けます。外部の都合（API のリクエスト仕様、スロットリング、エラー処理、キャッシュ等）はここに閉じ込めます。

3e. 依存方向のルール

層の間の矢印（依存）は内側に向かうのが原則:

infrastructure
→
port
→
application
→
domain

つまり:

  • domain は何にも依存しない（最も純粋）
  • application は domain と port を使う（infrastructure を知らない）
  • port は interface だけで依存ゼロ
  • infrastructure は port を実装する（内側に依存）

この方向を守ることで、「外側の都合で内側を書き換えたくなる」事態を防ぎます。

やってはいけない例: domain 層のコードが import { GmoRestClient } from '../infrastructure/...' と書く。これは「内側が外側を知る」ことになり、Clean Architecture 違反です。今回の案 A 〜 C が却下された理由の 1 つがこれ。

4. 値オブジェクトって何?

値オブジェクト（Value Object）って何?

ただの数値や文字列ではなく、意味と制約（ルール）を内側に持ったデータの箱。

❌ プリミティブ型で扱う場合

function calculateLot(
  balance: number,
  rate: number,
  marginRate: number,
): number { ... }
  • 呼び出し側で引数順を間違えても気づかない
  • 負の balance を渡してもそのまま動いてしまう
  • rate が NaN や 0 でもチェックされない
  • 単位（円? ドル?）が型で区別できない

✅ 値オブジェクトで扱う場合

function calculateLot(
  balance: Balance,
  rate: Rate,
  marginRate: MarginRate,
): Lot { ... }
  • 引数順を間違えると型エラーでコンパイル失敗
  • Balance は負にならない（生成時に弾く）
  • Rate は 0 や NaN を許さない
  • Money が通貨を持つので単位違いはありえない

値オブジェクトの 3 つの性質

  1. 不変（immutable） — 生成後に値を変えられない
  2. 同一性は属性値のみで決まる — Balance(1000円) と Balance(1000円) は「同じもの」
  3. 自己検証 — 生成時に不正な値を弾く

コード例

export class Balance {
  private constructor(private readonly amount: Money) {}

  static of(amount: Money): Balance {
    if (amount.isNegative()) {
      throw new Error('Balance は負になれません');  // ← 自己検証
    }
    return new Balance(amount);
  }

  // 操作は値オブジェクトのメソッドに生やす
  multipliedBy(ratio: Ratio): Money {
    return this.amount.times(ratio);
  }
}

このクラスは new Balance(...) で直接作れず、Balance.of(money) のファクトリ経由でしか生成できません。ファクトリが不変条件を検証するので、Balance が存在している時点で「非負の金額」が保証されます。

利点: バグを早期に捕捉できる。コードを読んだ時に型名から意味が伝わる（number より Balance のほうが何を表しているか明確）。

5. 命名の原則

名前はコードを読む人（他人、3 ヶ月後の自分）への最大のドキュメントです。役割が名前でわかれば、コメントを書かずに済みます。

プロジェクトに関わる全員（開発者・設計者・将来の保守者）が共通に使う言葉を、コードの中でも一貫して使います。

例: FX 業界では「ポジションのサイズをどう決めるか」を "Position Sizing" と呼びます。だから配分決定部品は PositionSizingService。これを LotCalculator や LotDecider と呼ぶと業界語彙から外れ、読んだ人が「何これ?」となります。

6a. 値オブジェクトと LotPolicy（図 A）

図 A: 値オブジェクトと LotPolicy — value-objects-and-lot-policy.drawio

domain 層の純粋なロット計算まわり。Balance / Money / Rate / Lot などの値オブジェクトと、LotDecisionInput（パラメータオブジェクト）、LotPolicy interface、実装 MaintenanceRatioBasedLotPolicy。

↑ ズーム・フルスクリーン・drawio で開く はツールバーから

LotPolicy domain (interface)

「入力パラメータ → Lot」を決めるドメインサービスの interface。

AllocationPolicy との違い

AllocationPolicyLotPolicy
役割 戦略間の配分比率を決める 個別の発注で何ロットかを決める
入力 検知シグナル + ポジション + 残高 通貨ペア + 残高 + レート + 維持率
出力 LotAllocation（比率の集合） Lot（単一の数値 with 不変条件）

AllocationPolicy が「全体の分配方針」で、LotPolicy が「個別の具体的な数量計算」を担います。

シグネチャ

interface LotPolicy {
  decide(input: LotDecisionInput): Lot;
}

LotDecisionInput はパラメータオブジェクト。引数が増えてもシグネチャが壊れない設計。

MaintenanceRatioBasedLotPolicy domain

「維持率が目標値になるよう Lot を決める」という具体的な計算ロジック。LotPolicy の実装。

なぜ長い名前?

旧称は MarginBasedLotPolicy（証拠金ベース）でしたが、「Margin ベース」は範囲が広すぎて意図が伝わりません。実態は「維持率（MaintenanceRatio）が目標値（例: 140%）になるようにロットを決める」という、より特定のロジック。

長くても意図が伝わる名前の方が、読んだ人が迷いません。もし将来別の計算ロジック（例: 固定 Lot、ATR ベース）が加わった時に、名前の違いで区別できます。

純粋ドメインサービス

このクラスは値オブジェクトだけを受け取って値オブジェクトを返す純粋関数のようなもの。DB も API も呼ばず、渡された入力だけで結果を計算します。

class MaintenanceRatioBasedLotPolicy implements LotPolicy {
  constructor(private readonly marginRate: MarginRate) {}

  decide(input: LotDecisionInput): Lot {
    // input の balance, rate, target（維持率）から計算
    // 外部状態（時刻、API、DB）は一切参照しない
    const capital = input.balance.amount();
    const rate = input.rate.value();
    const rawLot = Math.floor(
      capital / (input.target.value() * rate * this.marginRate.value())
    );
    return Lot.of(Math.min(Math.max(rawLot, 100), 500_000));
  }
}

純粋関数のようなドメインサービスは、テストがとても簡単です（モック不要、ただ値を渡すだけ）。

6b. AllocationPolicy と LotAllocation（図 B）

図 B: AllocationPolicy — allocation-policy.drawio

戦略間の配分判断。LotAllocation（配分結果）、AllocationPolicy（判断 interface）、初期実装 EqualWeight、将来候補 3 種（破線・薄色）。

↑ ズーム・フルスクリーン・drawio で開く はツールバーから

AllocationPolicy domain (interface)

「検知された戦略シグナル → 配分比率」を決めるドメインサービスの interface。

なぜ interface?

配分ロジックは将来複数のバージョンが並立する想定:

  • 初期: EqualWeightAllocationPolicy（等ウェイト）
  • 将来: FixedRatioAllocationPolicy（固定比率、#120）
  • 将来: RiskBudgetedAllocationPolicy（リスクパリティ、#121）
  • 将来: ConvictionWeightedAllocationPolicy（ConvictionScore ベース、#57）

使う側（PositionManager）は interface だけ知っていれば良く、実装を差し替えることで全く違うロジックに変えられます。

メソッド

interface AllocationPolicy {
  allocate(context: AllocationContext): LotAllocation;
}

入力は AllocationContext（検知シグナル + 現ポジション + 残高）、出力は LotAllocation（配分結果）。

LotAllocation domain (VO)

配分の「結果」を表す値オブジェクト。

なぜ AllocationPolicy と別?

「どう決めるか（ロジック）」と「決まった結果（値）」は別物。1 つのクラスに押し込むと、結果を使うだけのコード（ログ出力、UI 表示）までロジックに依存してしまう。分けることで、値だけを使い回せるようにしています。

不変条件

  • 配分比率の合計は必ず 1.0（または全戦略が抑制される場合 0.0）
  • 各比率は 0.0 〜 1.0
  • 一度生成されたら変更不可

Map を外に出さない

// ❌ 悪い例
class LotAllocation {
  constructor(public readonly ratios: Map<StrategyName, Ratio>) {}
}
// 外から ratios.set() で改変できてしまう

// ✅ 良い例
class LotAllocation {
  private constructor(
    private readonly ratios: ReadonlyMap<StrategyName, Ratio>
  ) {}
  ratioOf(s: StrategyName): Ratio { ... }
  isSuppressed(s: StrategyName): boolean { ... }
}

内部構造を外に見せず、振る舞い（ratioOf, isSuppressed）で問い合わせる形にすることで、改変を防ぎつつ表現力を確保します。

6c. エントリー発注の流れ（図 C）

図 C: エントリー発注フロー — composition-entry-flow.drawio

application 層を中心にした発注経路。PositionManager → AllocationPolicy → PositionSizingService → EntryQueue → EntryExecution → Broker。

↑ ズーム・フルスクリーン・drawio で開く はツールバーから

PositionManager application

複数戦略の同時稼働を束ねる司令塔。今回新しく作る中心部品。

なぜ必要?

今までは戦略が 1 つだったので、TradingSession が直接 EntryRule を呼んで EntryExecution に渡していました。複数戦略になると:

  • 複数 Rule が同時にシグナルを出した時の調整が必要
  • 戦略ごとに配分を決めないといけない
  • 発注順序を制御する必要がある

これらを 1 箇所にまとめる部品が PositionManager です。

依存するもの

  • AllocationPolicy（配分ロジック）
  • PositionSizingService（ロット計算）
  • EntryQueue（発注順序制御）
  • PositionRepository（ポジション取得・保存）

なぜ application 層?

ドメインの「判断」ではなく、「部品を組み合わせてユースケースを実現する組み立て役」だから。判断ロジックそのもの（どう配分するか）は domain（AllocationPolicy）に委譲し、PositionManager は組み立てに徹します。

PositionSizingService application

残高取得 → LotPolicy 呼び出しの組み立て役。

なぜこのクラスが必要?

LotPolicy は純粋ドメインサービスなので、残高の取り方を知らない。誰かが「残高を取ってきて、レートを取ってきて、LotPolicy に渡す」という組み立てをしないといけない。これを担うのが PositionSizingService。

コード骨子

class PositionSizingService {
  constructor(
    private readonly balancePort: BalancePort,
    private readonly ratePort: RatePort,
    private readonly lotPolicy: LotPolicy,
    private readonly fallbackBalance: Balance,  // CAPITAL 由来の予備値
  ) {}

  execute(pair: CurrencyPair): Lot {
    // ① 残高取得（失敗したら予備）
    const balance = this.balancePort.current() ?? this.fallbackBalance;
    // ② レート取得
    const rate = this.ratePort.currentOf(pair);
    // ③ 入力を束ねて LotPolicy に渡す
    return this.lotPolicy.decide(
      new LotDecisionInput(pair, balance, rate, MaintenanceRatio.of(1.4))
    );
  }
}

なぜ "Position Sizing"?

FX 業界用語。「ポジションサイズ（どれだけのロットで建てるか）をどう決めるかという判断」を総称する言葉。LotCalculator のような汎用語より意図が伝わります。

EntryQueue application

発注リクエストの順序制御。POST 1 秒 1 件制限の遵守と、古いシグナルの破棄。

なぜ独立した部品?

発注間隔の制御は PositionManager の関心事ではない。PositionManager は「配分を決めて発注指示を出す」までが仕事で、その指示を「いつ実際に API に送るか」は別の論点。分離することで、両者を独立してテストできます。

主要メソッド

  • enqueue(command: EntryCommand, submittedAt: Instant) — 発注指示を追加
  • drain() — 1 秒 1 件ペースで EntryExecution を呼ぶ（内部的に timer で動く）

古いシグナルの破棄

submittedAt から 3 秒経過した指示は Queue 内で破棄。FX は価格変動が速く、古い価格ベースのシグナルで発注するのは損な行為。

Broker port

発注（新規エントリー・決済）の契約を定める interface。

なぜ port 層にあるのか?

発注は外部（GMO API）との通信です。application 層が直接 GMO API を叩くと、GMO 以外のブローカーに切り替えたい時やテストで偽物の発注を使いたい時に困ります。そのため「発注という契約」だけを port で定義し、実装は infrastructure で提供する構造にします。

なぜ Broker という名前?

FX 業界で「発注を取り次ぐ業者」を Broker と呼ぶから。OrderClient や TradeApi のような名前だと業界語彙から外れます。

典型的な使われ方

class EntryExecution {
  constructor(
    private readonly broker: Broker,  // ← interface で受ける
    ...
  ) {}
  async execute(cmd: EntryCommand) {
    const result = await this.broker.placeEntry(cmd);
    ...
  }
}

BalancePort port

口座残高の取得を定める interface。

なぜ必要?

PositionSizingService がロット計算のために残高を知りたい。だが application 層から GMO API を直接叩くのは禁止（Clean Architecture 違反）。そのため「残高を取る契約」を port で定義。

2 つのメソッドを持つ理由

  • current(): Balance | null — 通常の問い合わせ。キャッシュされた値を返してよい。API 失敗時は null を返す
  • freshNow(): Balance — 発注直前の最新値取得。キャッシュを無視して API から直接取る

FX は価格変動が速いので、発注直前には最新残高で計算したい。一方で毎回 API を叩くとレートリミットに引っかかる。この使い分けを呼び出し側が明示的に選べるように 2 メソッドにしました。

なぜ名前が BalancePort? BalanceCache じゃない?

「Cache（キャッシュ）」は実装手段を名前に含めてしまっています。port の役割は「残高取得の契約」なので、実装がキャッシュを使っていても使っていなくても、名前は役割で呼ぶべき。キャッシュは GmoBalanceAdapter（infrastructure 側）の内部事情として隠蔽します。

PositionRepository port

ポジションの保存・取得を定める interface（Repository パターン）。

なぜ Repository?

ドメイン駆動設計における Repository パターン の適用。「ポジションは集合として扱えて、追加・取り出し・更新ができる」という抽象を提供します。

具体的には:

  • register(position) — 新規ポジションを追加
  • openPositions() — OPEN 状態のポジション全件を返す
  • findByPairAndStrategy(pair, strategy) — (pair, strategy_name) で検索
  • update(position) — 決済後の状態を反映

使う側（application 層）は「DB テーブル」を知らず、「ポジション集合」を扱うように書けます。

実装は PgPositionRepository

infrastructure 層の PgPositionRepository が PostgreSQL + Drizzle で実装。「(pair, strategy_name) の OPEN 部分ユニーク制約」もこの実装の責務。

GmoBalanceAdapter infrastructure

GMO FX API で口座残高を取得する実装。BalancePort の実装クラス。

なぜ名前が "Adapter"?

Ports & Adapters パターンの「Adapter」。port（契約）と外部システム（GMO API）の間を仲介する変換役、という意味。

内部の実装詳細

  • 5 秒の TTL キャッシュ — API のレートリミット対策
  • CAPITAL フォールバック — API 失敗時は環境変数値を返す
  • GMO API /private/v1/account/assets を呼ぶ
  • HMAC-SHA256 署名生成

なぜ名前にキャッシュを出さない?

キャッシュは実装手段で、役割ではないから。将来「キャッシュをやめる」「別の方法に変える」といった時に、クラス名を変えずに済みます。名前は "何をやるか"（Balance を取る） と "どこと話すか"（Gmo と） の 2 軸で付けています。

GmoBrokerAdapter infrastructure

GMO FX API で発注する実装。Broker の実装クラス。

役割

  • EntryCommand を GMO API のリクエスト形式に変換
  • HMAC-SHA256 署名
  • POST スロットリング（1 秒 1 件）
  • レスポンスを EntryResult に変換

EntryQueue との役割分担

POST 1 秒 1 件制限の遵守は、EntryQueue（上位の発注キュー）と GmoBrokerAdapter の両方に関わります。

  • EntryQueue: 「発注指示の順序」を制御（古いシグナル破棄など）
  • GmoBrokerAdapter: 「実際の API 呼び出しの間隔」を保証（失敗時のリトライ制御含む）

二重にしているのは冗長ではなく、責任を階層化しているため。EntryQueue は業務レベルの制御、Adapter は技術レベルの保証。

7. 典型的な「なぜ?」への答え

クラス図を見て湧く疑問の定番に、まとめて答えます。

なぜインターフェースを挟むのか? 直接クラスを使えばよくない?

テスト容易性と将来の差し替えのため。直接実装クラスに依存すると、テストで偽物を使えず、実装を変えたい時に使う側のコードも書き換えないといけません。interface を挟むと、使う側のコードを変えずに中身を差し替えられます。

なぜ LotPolicy と AllocationPolicy の両方がいるのか? 似たような名前で紛らわしい

責任範囲が違います。AllocationPolicy は「複数戦略の間での配分（どの戦略に何割?）」、LotPolicy は「個別発注の中での計算（Lot は何?）」。前者は戦略の司令塔レベル、後者は個別ロット決定レベル。紛らわしいけど、どちらも "Policy" という名前がふさわしい（判断方針のクラス）ので、修飾語で区別しています。

domain 層に興味があるクラスを置いたけど、外部 API も呼びたい。どうすれば?

domain 層では呼ばず、application 層に組み立て役を置いて、そこから port 経由で外部 API を呼びます。今回の PositionSizingService がまさにそれ。domain（LotPolicy）は純粋、application（PositionSizingService）が組み立て、infrastructure（GmoBalanceAdapter）が実際の通信、という三段構造です。

どのクラスから読み始めればいい?

PositionManager から。今回の中心部品で、他の主要クラスは全部 PositionManager が直接・間接に使います。そこから枝を辿って AllocationPolicy → LotAllocation、PositionSizingService → LotPolicy、EntryQueue → EntryExecution と読むと流れが掴みやすいです。

この設計、過剰じゃない? もっとシンプルにできないの?

感じ方はわかります。「1 つのクラスにまとめて書けば短いのに」と思うかもしれません。でも Luchida は「年末 200 万円運用のガチプロダクト」で、数年先まで保守・拡張する想定。今シンプルに書くと、将来「戦略を追加する」「配分ロジックを変える」「GMO 以外に対応する」「テストを書く」といった変更の度に、あちこち書き換える羽目になります。層を分けておくと、変更の影響範囲を限定できます。短期の書きやすさより長期の変更しやすさを優先した設計です。

付箋（黄色のメモ）の内容はどこまで信じていい?

クラス図内の付箋は「設計の意図」を表しています。コードよりもまず付箋を読むと、なぜそういう設計になっているかが掴みやすい。内容に疑問があれば、brief.md（設計ブリーフ）と突き合わせてください。brief.md が一次情報です。
